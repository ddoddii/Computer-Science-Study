### **1. 쿠키와 세션의 차이에 대해 설명해 주세요.**
쿠키는 사용자의 컴퓨터 또는 모바일기기에 저장되며, 만료 날짜가 지정되어 있어도 브라우저를 닫아도 계속해서 남아있을 수 있습니다. 세션은 서버에 저장되며, 만료 날짜가 지정되어 있지 않으면 브라우저를 닫으면 사라집니다. <br>
보안적인 측면에서도 쿠키는 클라이언트에 저장되어 있기 때문에 보안에 취약하고, 세션은 서버에 저장되어 있기 때문에 보안에 강합니다.
쿠키의 사용용도는 로그인 정보, 방문 기록 등이 있으며, 세션은 로그인 정보, 장바구니 등에 사용됩니다.
- **세션 방식의 로그인 과정에 대해 설명해 주세요.**
<br> 사용자가 로그인 ID와 비밀번호를 입력해 로그인 -> 웹서버는 사용자가 입력한 정보에 데이터베이스에 저장된 정보와 비교하여 인증 ->
인증이 성공하면 서버는 세션을 생성하고 세션 ID를 생성 -> 서버는 세션 ID를 쿠키에 담아 사용자의 브라우저에 전송<br> -> 사용자의 브라우저는 쿠키에 저장된 세션 ID를 사용하여 서버에 요청
-> 서버는 쿠키에 저장된 세선ID를 확인하여 사용자 인증을 수행 -> 인증이 성공하면 사용자는 로그인되고 웹사이트를 이용
- **HTTP의 특성인 Stateless에 대해 설명해 주세요.**
<br> HTTP의 Stateless 특성은 서버가 클라이언트의 이전 요청 정보를 기억하지 않는다는 것을 의미합니다. 즉, 각 요청은 서로 독립적으로 처리됩니다. <br>
장점으로는 서버가 클라이언트의 상태를 기억할 필요가 없기때문에 많은 클라이언트를 처리할수있다는것, 그리고 서버구현이 간단하고 오류가능성이 적으며, 서버에 상태에 영향을 받는 오류가 발생할 가능성이 낮을수있다는겁니다.
<br> 단점으로는 사용자의 세션정보를 유지하려면 쿠키나 세션을 통해 추가적인 메커니즘이 필요하고 관리해야한다는점, 동일한 정보를 반복적으로 전송할수도있어서 효율성이 떨어집니다.
- **Stateless의 의미를 살펴보면, 세션은 적절하지 않은 인증 방법 아닌가요?**
세션은 Stateless 특성을 가진 HTTP에서 사용하기에는 완벽하지 않은 인증 방법이지만, 편리함,안정성 등의 장점 때문에 많이 사용됩니다. 세션을 사용할 때는 보안과 세션 관리에 주의해야 합니다.
- **규모가 커져 서버가 여러 개가 된다면, 세션을 어떻게 관리할 수 있을까요?**
1. 모든 서버가 동일한 세션 정보에 접근할수있도록 공유하는 방식
2. 세션정보를 데이터베이스에 저장하여 세션정보를 불러오는 방식
3. 자주 사용되는 세션정보를 캐싱하여 서버 부하를 줄이면서 불러오는 방식
4. 사용자를 특정 서버에 고정시키는 방식(스티키 세션)
5. 로드 밸런서를 사용하여 사용자 요청에 특정 서버를 고정시키는 방식
6. 사용자의 IP 주소를 기반으로 고정시키는 방식
7. 사용자 브라우저에 쿠키를 저장하여 특정 서버에 고정시키는 방식
8. 세션 클러스터링을 사용하여 여러 서버에 세션 정보를 분산하는 방식
- **토큰 기반 방식은 어떤 장점이 있을까요?**
1. Stateless 특성을 가진 HTTP에서 사용하기에 적합합니다.
2. 서버의 부하를 줄일 수 있습니다.
3. 사용자의 정보를 저장하지 않기 때문에 보안성과 서버의 확장성이 높습니다.
4. 세션 정보를 서버에서 조회할 필요가 없기때문에 응답속도가 빠릅니다. 그리고 토큰을 캐싱하여 성능또한 더 빠르게 할수 있습니다.
5. 서버 장애 발생시에도 다른 서버에서 사용자 정보를 확인할수있습니다.
### **2. HTTP 응답코드에 대해 설명해 주세요.**
1. 2xx(성공)
   1. 200 (OK) : 요청이 성공했을 때 반환합니다.
   2. 201 (Created) : 요청이 성공하고 새로운 리소스가 생성되었을 때 반환합니다.
   3. 204 (No Content) : 요청이 성공했지만 응답 본문에 정보가 없을 때 반환합니다.
2. 3xx(리다이렉션)
   1. 301 (Moved Permanently) : 요청한 리소스가 새로운 URL로 옮겨졌을 때 반환합니다.
   2. 302 (Found) : 요청한 리소스가 임시적으로 새로운 URL로 옮겨졌을 때 반환합니다.
   3. 304 (Not Modified) : 요청한 리소스가 수정되지 않았을 때 반환합니다.
3. 4xx (클라이언트 오류)
   1. 400 (Bad Request) : 잘못된 요청이 있을 때 반환합니다.
   2. 401 (Unauthorized) : 인증이 필요한 페이지에 인증되지 않은 사용자가 접근했을 때 반환합니다.
   3. 403 (Forbidden) : 요청이 서버에서 거부되었을 때 반환합니다.
   4. 404 (Not Found) : 요청한 리소스가 서버에 없을 때 반환합니다.
   5. 405 (Method Not Allowed) : 요청한 메소드가 허용되지 않을 때 반환합니다.
4. 5xx (서버 오류)
   1. 500 (Internal Server Error) : 서버에서 요청을 처리하는 동안 오류가 발생했을 때 반환합니다.
   2. 501 (Not Implemented) : 서버가 요청을 처리할 수 없을 때 반환합니다.
   3. 502 (Bad Gateway) : 서버가 게이트웨이나 프록시 역할을 하고 있을 때, 다른 서버로부터 잘못된 응답을 받았을 때 반환합니다.
   4. 503 (Service Unavailable) : 서버가 요청을 처리할 수 없을 때 반환합니다.
   5. 504 (Gateway Timeout) : 서버가 게이트웨이나 프록시 역할을 하고 있을 때, 다른 서버로부터 응답을 받지 못했을 때 반환합니다.
- **401 (Unauthorized) 와 403 (Forbidden)은 의미적으로 어떤 차이가 있나요?**
<br>401과 403 모두 클라이언트가 요청한 리소스에 접근할 수 없는것을 나타내는 HTTP 응답코드지만 인증여부에 차이가 있습니다. <br>
401은 클라이언트 요청에 대한 인증 정보를 제공하지 않았거나 제공된 인증 정보가 유효하지 않은 경우<br>
403은 클라이언트는 인증되었지만 요청된 리소스에 대한 접근 권한이 없는 경우입니다.
- **200 (ok) 와 201 (created) 의 차이에 대해 설명해 주세요.**
200 OK은 201 Created는 모두 클라이언트의 요청이 성공적으로 처리되었음을 나타내는 HTTP 응답 코드이지만, 리소스 생성 여부에서 차이가 있습니다.
<br> 200은 클라이언트의 요청이 성공적으로 처리되었고 기존 리소스에 대한 응답이며, 201은 클라이언트의 요청이 성공적으로 처리되었고 새로운 리소스가 생성되었음을 의미합니다.
<br> 200 OK은 일반적으로 GET, PUST, DELETE 요청에 대한 응답으로 사용되며, 201은 POST 요청으로 응답됩니다.
- **필요하다면 저희가 직접 응답코드를 정의해서 사용할 수 있을까요? 예를 들어 285번 처럼요.**
<br>가능합니다. 다만 HTTP 표준을 준수해야하며, 코드 의미를 명확하게 문서화하고 공개해야합니다. 다만 정말 특별한 상황이 아니라면 사용자 정의 코드를 사용하는 것은 피하는것이 좋습니다.
사용자 정의 코드는 불필요한 복잡성을 증가시키고 유지 관리에도 어려움을 초래하 수 있습니다.
